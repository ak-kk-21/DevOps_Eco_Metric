01

Project overview

Developers and engineers currently have no quick, accessible way to estimate how much carbon their code or cloud infrastructure produces. Running a large language model training job, a nightly ETL pipeline, or even a fleet of microservices all have measurable environmental costs — but this data is invisible during the design phase.

The Eco-Metric API solves this by exposing a simple HTTP interface. You send it a description of your computing workload — hardware type, hours of use, cloud region — and it returns an estimated CO₂ output in grams, alongside human-readable equivalents like "driving 3.1 km" or "charging 54 smartphones".

The system is stateless: no database, no user accounts, no persistent state. Data lives in two JSON fixture files baked into the container. This keeps the architecture simple, the tests deterministic, and the deployment bulletproof.

This project is intentionally scoped to be simple to build and simple to explain. Every architectural decision prioritises testability, clean CI output, and confident Q&A answers over feature breadth.

05

Calculation model

The calculation uses the standard energy-to-emissions formula from green computing research. All variables come from the request body and the two fixture JSON files.

energy_kwh = (tdp_watts × utilization × hours) / 1000
co2_grams = energy_kwh × carbon_intensity (gCO₂/kWh)
Where TDP (Thermal Design Power) is the maximum power draw of the hardware in watts.

Example: an NVIDIA A100 GPU (400W TDP) running at 85% utilisation for 2.5 hours in us-east-1 (386 gCO₂/kWh):

energy = (400 × 0.85 × 2.5) / 1000 = 0.85 kWh
co2 = 0.85 × 386 = 328.1 grams CO₂
The equivalent_to field is computed by dividing co2_grams by 170 (average gCO₂/km for a petrol car). This one field transforms an abstract number into something a professor or audience will immediately understand.
06

CI/CD pipeline

Every push to any branch triggers the first three stages. Stages four and five only run on merges to main. Branch protection on main requires all checks to be green before a merge is allowed.

1
Install dependencies via Poetry
Runs on every push. Sets up the Python virtualenv using the lock file for deterministic builds.
2
Run Pytest with coverage
Executes all tests in /tests. Fails the pipeline if any test fails. Generates XML coverage report.
3
SonarCloud scan
Ingests the coverage XML. Fails if coverage drops below 80% or if critical code smells are introduced. Blocks the PR.
4
Build and push Docker image
Only on merge to main. Builds the image and pushes to Docker Hub tagged with the commit SHA and latest.
5
Render auto-deploy
Render watches Docker Hub and pulls the new image automatically. Live URL is updated within ~60 seconds.
07

Testing strategy

Tests are split into two files. Unit tests in test_calculator.py cover the pure math functions — they run with no network or I/O, are fast, and are trivial to write. Integration tests in test_routes.py use FastAPI's test client to hit the actual endpoints.

Example unit test (test_calculator.py)
def test_energy_calculation_known_values():
  # A100 GPU at 85% utilisation for 2.5 hours
  result = calculate_energy_kwh(
    tdp_watts=400,
    utilization=0.85,
    hours=2.5
  )
  assert round(result, 3) == 0.85

def test_zero_utilization_returns_zero():
  result = calculate_energy_kwh(tdp_watts=400, utilization=0.0, hours=10)
  assert result == 0.0

def test_co2_proportional_to_intensity():
  low = calculate_co2_grams(energy_kwh=1.0, intensity=100)
  high = calculate_co2_grams(energy_kwh=1.0, intensity=400)
  assert high == low * 4

The six seeded test scenarios cover: GPU training job, CPU batch job, idle server, unknown hardware (422 error), unknown region (422 error), and a zero-hour edge case. Together these trivially clear the 80% coverage threshold.
